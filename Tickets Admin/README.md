# Tickets Admin automation

Готовая заготовка Google Apps Script для автоматизации заявок в Slack-канале `#tickets-admin` по ТЗ.

## Что реализовано

- `/ticket` и Global Shortcut `Создать тикет` открывают Slack Modal.
- `/support` открывает отдельную форму для задач лида техподдержки: `Мед учреждение`, `Что нужно сделать`, приоритет и фото.
- `/add` открывает Slack Modal для учета выполненных вне Slack задач; сохранять такие задачи могут только активные админы.
- Форма содержит обязательные поля: тип заявки из справочника, регион/ОП/ID клиники, приоритет, детали, а также опциональное поле для фото.
- После отправки бот публикует структурированную карточку тикета в `#tickets-admin` с максимально коротким ID по порядку: `T1`, `T2`, `T3`.
- Цветная статусная полоса сохраняется, а кнопки статуса стоят сразу после блока `Детали`, чтобы они были выше `Show more / Show less`.
- ПИНФЛ, серии/номера паспортов, номера телефонов Узбекистана, метрики вида `I-TN 234576` и похожие записи автоматически удаляются из публичных полей тикета до публикации.
- Попытки отправить персональные данные фиксируются в отдельном листе `Personal Data Audit` без сохранения самих персональных данных.
- Создание тикета и смена статусов выполняются через постоянную минутную очередь Apps Script, поэтому `doPost` не создает trigger во время Slack-запроса и отвечает быстрее 3 секунд.
- Все активные админы из листа `Admins` могут взять новую заявку в работу.
- После `Взять в работу` завершить или отклонить тикет может только тот админ, который взял его в работу; остальные админы получат предупреждение, а статус не изменится.
- Статус в Slack-карточке вынесен в крупную отдельную строку с цветным маркером, при этом цветная статусная полоса слева сохраняется.
- Нижний служебный статусный текст под кнопками не выводится.
- Повторные нажатия на submit/action-кнопки дедуплицируются, чтобы не создавать дубли в Slack.
- При отказе админ обязан указать причину в отдельном Slack modal; причина сохраняется в Google Sheets и отправляется автору в тред.
- Инициатор получает уведомления в треде под карточкой, фото из формы автоматически отправляются туда же. Если персональные данные были удалены, бот отдельно уведомляет автора в треде.
- Все тикеты пишутся в Google Sheets, события статусов пишутся в лист `Events`.
- Вся техподдержка хранится в отдельной Google-таблице: `Tech Support Tickets`, `Tech Support Events`, `Tech Support Admins`, `Tech Support Reports`, `Tech Support Dashboard` и свой `Personal Data Audit`.
- `setup()` создает лист `Dashboard` с формулами и графиками по объему, типам, регионам, админам и SLA.
- Каждый понедельник в 09:00 по Ташкенту бот отправляет недельный отчет в канал.
- `/report 05.05.2026 06.05.2026` показывает статистику за период.
- `/report @admin 05.05.2026 06.05.2026` показывает статистику конкретного администратора.
- `/support-report 05.05.2026 06.05.2026` и `/support-report @admin 05.05.2026 06.05.2026` показывают такую же статистику только по техподдержке.
- Вне Slack задачи из `/add` сохраняются в отдельный лист `Вне Slack задачи`, а аналитика по ним выводится на отдельный лист `Вне Slack аналитика`.
- Лист `Reports` в Google Sheets красиво оформляет последний сформированный отчет.
- Лист `Employee Report` позволяет выбрать сотрудника и дату, чтобы увидеть выполненные Slack-тикеты и вне Slack задачи.
- Лист `Типы заявок` управляет выпадающим списком типов в Slack форме без правки кода.
- В листе `Типы заявок` можно назначить ответственного другого отдела и текст: после создания такого тикета бот автоматически отметит его в треде.
- Если новый тикет не взят в работу 3 дня, бот отмечает случайного активного админа в треде; если тикет в работе больше 1 дня, бот отмечает исполнителя.

## Файлы

- `Code.gs` - основной код Google Apps Script.
- `appsscript.json` - манифест Apps Script со scopes и таймзоной.
- `slack-fast-relay-worker.js` - быстрый Cloudflare Worker для Slack ACK меньше 3 секунд.
- `wrangler.toml.example` - пример конфига Cloudflare Worker.
- `slack-app-manifest.example.yml` - пример Slack manifest.
- `README.md` - инструкция по запуску.

## Настройка Google Apps Script

1. Создайте проект в Google Apps Script.
2. Перенесите в проект `Code.gs` и `appsscript.json`.
3. В `Project Settings -> Script Properties` добавьте:

| Property | Значение |
| --- | --- |
| `SLACK_BOT_TOKEN` | Bot User OAuth Token, например `xoxb-...` |
| `SLACK_TICKETS_CHANNEL_ID` | ID канала `#tickets-admin`, например `C0123456789` |
| `SLACK_REPORT_CHANNEL_ID` | Необязательно. Канал для еженедельных отчетов. Если пусто, используется `SLACK_TICKETS_CHANNEL_ID` |
| `SLACK_SUPPORT_CHANNEL_ID` | ID канала для тикетов лида техподдержки |
| `SLACK_SUPPORT_REPORT_CHANNEL_ID` | Необязательно. Канал для отчетов техподдержки. Если пусто, используется `SLACK_SUPPORT_CHANNEL_ID` |
| `ADMIN_USER_IDS` | Необязательно. Запасной список Slack ID через запятую, если лист `Admins` пустой |
| `SUPPORT_ADMIN_USER_IDS` | Необязательно. Запасной список Slack ID для техподдержки, если лист `Tech Support Admins` пустой |
| `SPREADSHEET_ID` | Необязательно. Если пусто, `setup()` создаст таблицу |
| `SUPPORT_SPREADSHEET_ID` | Необязательно. ID отдельной таблицы техподдержки. Если пусто, `setup()` создаст ее и сохранит ID автоматически |
| `SLACK_VERIFICATION_TOKEN` | Необязательно. Legacy token Slack, если доступен |
| `RELAY_SHARED_SECRET` | Необязательно. Общий секрет между Worker и Apps Script |
| `DIRECT_MODAL_OPENING` | Не заполняйте. `true` нужен только для старого прямого режима без Worker |
| `REPORT_CUTOFF_AT` | Необязательно. Дата начала новой статистики; `setup()` заполнит автоматически, если пусто |
| `WORKER_URL` | Рекомендуется. URL Cloudflare Worker, чтобы Apps Script заранее обновлял быстрый кэш типов заявок |

Если используете Worker, лучше оставить `SLACK_VERIFICATION_TOKEN` пустым и включить `RELAY_SHARED_SECRET` / `APPS_SCRIPT_RELAY_SECRET`. Slack-подпись проверяет Worker, а Apps Script принимает только запросы от relay.

4. Запустите функцию `setup()` из редактора Apps Script и выдайте разрешения.
5. Проверьте созданную Google-таблицу: доступ к ней оставьте только Лиду и администраторам.
6. Нажмите `Deploy -> New deployment -> Web app`.
7. Укажите `Execute as: Me` и `Who has access: Anyone`, затем скопируйте Web App URL.

## Администраторы

После запуска `setup()` появится отдельный лист `Admins`. Чтобы добавить администратора, добавьте строку:

| Slack User ID | Name | Role | Active | Notes |
| --- | --- | --- | --- | --- |
| `U1234567890` | Sardor | Admin | `TRUE` | может брать тикеты |

- `Slack User ID` обязателен. Это именно Slack ID вида `U...`, не display name.
- `Active` должен быть `TRUE`, `да`, `yes` или `1`. Если поставить `FALSE`, `нет`, `no`, `0`, `disabled`, пользователь потеряет право нажимать кнопки.
- Проверка админов кэшируется на 60 секунд, поэтому новое право может примениться не мгновенно, а примерно в течение минуты.
- Если лист `Admins` пустой, код использует запасной `ADMIN_USER_IDS` из Script Properties.
- Для техподдержки используется лист `Tech Support Admins` в отдельной таблице техподдержки с теми же колонками. Только пользователи из этого листа или из `SUPPORT_ADMIN_USER_IDS` могут нажимать кнопки в тикетах техподдержки.

## Типы заявок

После запуска `setup()` появится лист `Типы заявок`. Каждая активная строка попадает в выпадающий список поля `Тип заявки` в Slack. При необходимости эта же строка запускает автоматическое упоминание другого отдела в треде заявки:

| Тип заявки | Активен | Порядок | Примечание | Ответственный Slack ID | Сообщение в тред |
| --- | --- | --- | --- | --- | --- |
| `Включить склад 2.0` | `TRUE` | `1` | склад | `U1234567890` | `Пожалуйста, подтвердите подключение склада для этой заявки.` |

- Чтобы добавить тип, добавьте новую строку.
- Порядок колонок в листе не важен: код ориентируется по названиям заголовков, поэтому `Порядок` можно оставить перед `Активен`.
- Чтобы временно скрыть тип из Slack формы, поставьте `Активен` = `FALSE`.
- Чтобы отметить ответственного, заполните `Ответственный Slack ID` идентификатором Slack вида `U...`; можно указать несколько ID через запятую.
- Если заполнен ответственный, бот после создания карточки отправит в ее тред упоминание и текст из `Сообщение в тред`. Если текст пустой, используется стандартная просьба подключиться к заявке.
- Уведомление отправляется один раз и не меняет право админа брать тикет в работу или завершать его.
- Worker кэширует список примерно на 60 секунд, поэтому изменение может появиться в Slack не мгновенно.

## Быстрый Slack relay

Slack требует ответить на slash command/interactivity за 3 секунды. Google Apps Script иногда стартует дольше, а для открытия modal еще нужен вызов Slack API `views.open`. Поэтому Slack Request URL должен указывать не на Apps Script напрямую, а на быстрый relay.

Вариант через Cloudflare Workers:

1. Создайте Cloudflare Worker.
2. Загрузите в него `slack-fast-relay-worker.js`.
3. В переменных Worker задайте:

| Variable | Значение |
| --- | --- |
| `APPS_SCRIPT_URL` | Web App URL вашего Apps Script |
| `SLACK_SIGNING_SECRET` | Signing Secret из Slack App, лучше как secret |
| `SLACK_BOT_TOKEN` | Bot User OAuth Token `xoxb-...`, лучше как secret |
| `APPS_SCRIPT_RELAY_SECRET` | Необязательно. То же значение, что `RELAY_SHARED_SECRET` в Apps Script |
| `ADMIN_USER_IDS` | Необязательно. Slack ID админов через запятую для быстрого отказа на `/add` до открытия формы |
| `TICKET_TYPE_OPTIONS` | Необязательно. Запасной список типов через запятую или с новой строки, если кэш еще пуст |

Если используете Wrangler:

```bash
copy wrangler.toml.example wrangler.toml
wrangler secret put SLACK_SIGNING_SECRET
wrangler secret put SLACK_BOT_TOKEN
wrangler secret put APPS_SCRIPT_RELAY_SECRET
wrangler deploy
```

Чтобы `/add` не открывался у не-админов еще на уровне Worker, заполните `ADMIN_USER_IDS` в `wrangler.toml` теми же Slack ID, что активны в листе `Admins`.

После deploy скопируйте Worker URL, например `https://tickets-admin-slack-relay.your-subdomain.workers.dev`.

## Настройка Slack App

1. Создайте Slack App.
2. В `OAuth & Permissions` добавьте Bot Token Scopes:
   - `chat:write`
   - `commands`
   - `files:read`
3. Установите приложение в workspace.
4. Пригласите бота в `#tickets-admin`.
5. В `Slash Commands` создайте команду:
   - Command: `/ticket`
   - Request URL: Worker URL, не Apps Script URL
6. В `Slash Commands` создайте команду:
   - Command: `/support`
   - Request URL: тот же Worker URL
   - Usage hint: пусто
7. В `Slash Commands` создайте команду:
   - Command: `/report`
   - Request URL: тот же Worker URL
   - Usage hint: `05.05.2026 06.05.2026` или `@admin 05.05.2026 06.05.2026`
8. В `Slash Commands` создайте команду:
   - Command: `/support-report`
   - Request URL: тот же Worker URL
   - Usage hint: `05.05.2026 06.05.2026` или `@admin 05.05.2026 06.05.2026`
9. В `Slash Commands` создайте команду:
   - Command: `/add`
   - Request URL: тот же Worker URL
   - Usage hint: пусто
10. В `Interactivity & Shortcuts` включите interactivity:
   - Request URL: тот же Worker URL
   - Options Load URL: тот же Worker URL
11. Там же создайте Global Shortcut:
   - Name: `Создать тикет`
   - Callback ID: `ticket_create_shortcut`
12. Переустановите приложение после изменения scopes, slash commands или shortcuts.

Если `/report` уже занята другим Slack App, Slack может отправлять команду не этому боту. В таком случае удалите старую команду у другого приложения или переименуйте команду в `slack-fast-relay-worker.js` и `slack-app-manifest.example.yml`, например на `/tickets-report`.

Если в поле `Тип заявки` показывается запасной список, значит быстрый кэш Worker еще пуст или не обновился. Проверьте, что задеплоены свежие `Code.gs` и `slack-fast-relay-worker.js`, `Options Load URL` указывает на Worker URL, `WORKER_URL` в Apps Script указывает на Worker URL, а `APPS_SCRIPT_RELAY_SECRET` в Worker совпадает с `RELAY_SHARED_SECRET` в Apps Script. После изменения листа `Типы заявок` Apps Script пушит новый список в Worker, а Worker отдает его Slack уже без ожидания Google.

## Проверка сценария

1. Пользователь вызывает `/ticket` или shortcut `Создать тикет`.
2. Заполняет форму и отправляет.
3. Фоновый обработчик публикует карточку со статусом `Новая` в `#tickets-admin`; обычно это занимает до 1 минуты.
4. Если в форме были фото, они появляются отдельным сообщением в треде карточки.
5. Пользователь не из листа `Admins` нажимает кнопку и получает эфемеральную ошибку.
6. Админ нажимает `Взять в работу`; карточка обновляется, появляется `Выполнено` и `Отклонить`.
7. В треде появляется уведомление для инициатора.
8. Если другой админ нажмет `Выполнено` или `Отклонить`, система покажет предупреждение и не изменит статус.
9. Если исполнитель нажимает `Отклонить`, Slack открывает окно причины отказа. Без причины отказ не применяется.
10. Исполнитель нажимает `Выполнено`; карточка становится зеленой, в таблице заполняется `Date Completed` и `Resolution Time`.
11. В листе `Dashboard` автоматически обновляются таблицы и графики.
12. Пользователь не из листа `Admins` вызывает `/add` и не может сохранить вне Slack задачу; если в Worker заполнен `ADMIN_USER_IDS`, форма даже не откроется.

Для техподдержки сценарий такой же, но пользователь вызывает `/support`, карточка уходит в канал `SLACK_SUPPORT_CHANNEL_ID`, ID идет отдельно как `TS1`, `TS2`, а кнопки могут нажимать только активные пользователи из `Tech Support Admins`.

## Аудит персональных данных

Если пользователь указал ПИНФЛ, серию и номер паспорта, номер телефона, метрику вида `I-TN 234576` или похожие персональные данные в тикете или `/add`, система:

- удаляет эти данные из публичной карточки тикета или текста вне Slack задачи до публикации/сохранения;
- пишет автору в тред: `Если эти сведения нужны для выполнения заявки, передайте их администратору только через личные чаты Slack.`;
- добавляет запись в лист `Personal Data Audit`.

В `Personal Data Audit` сохраняются только дата, тикет или источник `Outside Slack`, автор и список очищенных полей. Сами персональные данные не сохраняются.

## Отчеты

- Еженедельный отчет отправляется каждый понедельник в 09:00 по таймзоне проекта Apps Script `Asia/Tashkent`.
- Еженедельный отчет считается за прошлую полную неделю: с понедельника 00:00 до воскресенья 23:59:59.
- `/report 05.05.2026 06.05.2026` считает период включительно: с 00:00 5 мая до 23:59:59 6 мая.
- `/report @admin 05.05.2026 06.05.2026` фильтрует статистику по исполнителю тикета (`Assignee` / `Assignee Name`).
- `/support-report 05.05.2026 06.05.2026` считает тот же период, но только по листу `Tech Support Tickets`.
- `/support-report @admin 05.05.2026 06.05.2026` фильтрует статистику по исполнителю из `Tech Support Admins`.
- В Slack показываются только данные по Slack-тикетам: создано, взято в работу, выполнено, отклонено, в работе сейчас, среднее время выполнения, общее время выполнения и среднее время реакции.
- Время выполнения/решения считается от момента `Взять в работу` (`Date Accepted`) до закрытия тикета (`Date Completed`), а не от момента создания заявки.
- В Google Sheets лист `Reports` перезаписывается последним сформированным отчетом по Slack-тикетам и содержит summary-блоки, таблицы и графики.
- В отдельной Google-таблице техподдержки лист `Tech Support Reports` перезаписывается последним отчетом техподдержки; данные не попадают в основной `Reports`.
- В Google Sheets лист `Вне Slack аналитика` перезаписывается тем же периодом отчета и показывает вне Slack задачи отдельно: итоги, сотрудников, детали и график.
- В Google Sheets лист `Employee Report` содержит выпадающий список сотрудников из `Admins` и дату; ниже выводятся выполненные Slack-тикеты и вне Slack задачи за выбранный день.
- Старые тикеты, созданные до даты текущего обновления, не входят в новую статистику. Дата отсечения хранится в Script Property `REPORT_CUTOFF_AT`; отчетность считает ее с 00:00 этого дня, чтобы тикеты текущего дня не пропадали из отчета.

## После обновления кода

1. Перенесите обновленные `Code.gs` и `appsscript.json` в Apps Script.
2. В Script Properties добавьте `WORKER_URL` со значением Worker URL, `SLACK_SUPPORT_CHANNEL_ID` и при необходимости `SLACK_SUPPORT_REPORT_CHANNEL_ID` / `SUPPORT_ADMIN_USER_IDS`. `SUPPORT_SPREADSHEET_ID` можно оставить пустым: таблица техподдержки создастся автоматически.
3. Запустите `setup()` еще раз. Основная таблица сохранит `Admins` / `Reports` / `Personal Data Audit` / `Вне Slack задачи` / `Вне Slack аналитика` / `Employee Report` / `Типы заявок`; в справочник добавятся колонки ответственного и текста, а в тикеты - отметка отправки уведомления. Отдельная таблица техподдержки получит `Tech Support Tickets` / `Tech Support Events` / `Tech Support Admins` / `Tech Support Reports` / `Tech Support Dashboard` / `Personal Data Audit`.
4. При первом запуске обновления `setup()` перенесет уже существующие вкладки `Tech Support ...` из основной таблицы в отдельную таблицу и перенесет туда строки аудита тикетов `TS...`. Ссылка на обе таблицы возвращается результатом функции `setup()`.
5. Обновите deployment Web App.
6. Задеплойте `slack-fast-relay-worker.js`.
7. В Slack App добавьте scope `files:read` и переустановите приложение в workspace.
8. В Slack App добавьте slash commands `/support`, `/support-report`, `/add` и замените Request URL для `/ticket`, `/support`, `/add`, `/report`, `/support-report`, Interactivity Request URL и Options Load URL на Worker URL.

Если в Executions всё еще появляется `doPost` длительностью больше 3 секунд при `/ticket`, значит Slack всё еще отправляет `/ticket` прямо в Apps Script. В `Slash Commands` и `Interactivity & Shortcuts` должен стоять Worker URL, а не `script.google.com`.

Slack `file_input` принимает файлы до 10 MB. В текущей форме разрешены фото `jpg`, `jpeg`, `png`, `gif`, максимум 5 файлов.

## Важное по безопасности

Apps Script Web Apps получают параметры и тело POST-запроса через объект `e`, но не дают надежного доступа к Slack headers. Поэтому прямая проверка Slack Signing Secret внутри чистого Apps Script ограничена. В коде поддержан `SLACK_VERIFICATION_TOKEN`, если он есть в вашем Slack-приложении. Для строгой проверки подписи поставьте перед Apps Script тонкий proxy на Cloud Run/API Gateway, который валидирует `X-Slack-Signature` и `X-Slack-Request-Timestamp`.
